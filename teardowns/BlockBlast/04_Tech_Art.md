# Block Blast 規格書 - 04. 技術與美術渲染
> 分析基礎：防護級渲染優化與動態物理演出
> 負責人：Tech Art AI

## 1. 2D 陣列的零殘酷渲染策略 (Zero-Overdraw Canvas)
### 1.1 偽立體化輕量色塊 (Pseudo-3D Light Rendering)
消除遊戲最嚴重的效能問題是不斷疊加的發光特效與滿版粒子，導致了「過度繪製 (Overdraw)」進而讓中低階手機嚴重發熱降頻。《Block Blast》從美術風格立項的源頭就採取極端防範：
- 它徹底捨棄了傳統厚重的 3D 發光水晶或帶有折射反光的寶石渲染模型。
- 改用具有溫順視覺手感的「原木 / 素面軟糖橡膠」材質。所有方塊模型都是純粹的 2D 元素，僅依賴輕量的 `box-shadow` 與 `linear-gradient` 添加高光與陰影倒角。
- 這保證了即使在擁有超過五年服役歷史的廉價 Android 開發測試機上，依然能保持絕對 60FPS 的絲質拖曳手感。

### 1.2 嚴格封鎖粒子上限 (Particle Masking Limit)
當單排方塊爆破時噴發的木屑或星光碎片，一律強制導入「物件池 (Object Pooling)」循環技術。
- 全畫面同框最大生成粒子數上限被鐵腕封鎖在 12~15 顆以內。
- 若行銷面需要更華麗的全版面破壞感受，則利用單一張預先烘焙渲染好的、帶有 Alpha Fade（漸隱）與動態模糊的 Sprite 巨大靜態貼圖，在前景極快地閃現並漸退蒙混過去，絕對禁止透過堆疊數十顆真粒子來運算發光層。

## 2. 極致物理感與多肉動畫 (Juicy Dynamic Animations)
### 2.1 擠壓與拉伸錯覺 (Squash and Stretch Delusion)
為彌補純量 2D 色塊的視覺單調，當玩家將方塊「抵達網格底部並鬆開」的那一個關鍵畫格，渲染引擎會對該方塊模型施加迪士尼經典動畫的 12 原則：
- 該方塊會立刻在 Y 軸（縱向）被壓縮 15%、且 X 軸（橫向）被擠扁膨脹 10%。
- 緊接著在 0.2 秒的 Spring（阻尼彈簧）參數運動曲線內彈跳回復原形狀。
- 這極短且誇張的一瞬間，能巧妙欺騙大腦，賦予這張 2D 虛擬色塊圖片難以言喻的「緊實彈性」、「重量感」以及「入槽卡榫聲」的錯覺。

### 2.2 漣漪波的骨牌效應 (Ripple Effect Spreading)
當玩家完成數條整列滿格的連鎖消除 (Combo Clear) 條件時，最忌諱的做法是將整列方塊同時套用 `opacity: 0` 讓它們死板地同步隱形，這會顯得極度廉價且缺乏物理力度。
- 正確做作法是：導入微小的極座標延遲擴散算式。
- 以放入最後一塊方塊的座標為「引爆震央點」，向四周與橫向兩端呈 0.05 秒的階梯微距延遲依次炸開。這能造就一種連環骨牌的「漣漪衝擊波爆破 (Ripple Impact)」，在達成多行大滿貫清台時的爽感無可匹敵。

## 3. 前後端渲染分層解耦 (Frontend State Decoupling)
### 3.1 邏輯腦運算先行，視覺手腳延遲跟上
這套遊戲最底層的核心 8x8 必須設計成極端輕量的 `Array[64]` 緩存資料陣列。
- 當玩家鬆開滑鼠 / 手指釋放方塊時，核心陣列的「非同步計算模組」必須優先在 `1 ms (毫秒)` 內跑完判定結果（`canPlace = true` 並且立刻遞迴結算後續引發的相消爆破範圍）。
- 算完之後，主程式才將「動畫播放列表與座標」丟給最外層的視覺層（CSS Transition 或 React DOM Render），放任畫面慢悠悠地去依序跑完後續 0.3 秒的方塊入槽與 0.5 秒的骨牌爆炸特效。
- 使「遊戲真實驗證狀態」永遠比畫面快一步，徹底斬斷因為畫面太華麗導致掉幀、進而影響玩家下一個拖拉輸入點擊掉幀被漏算的嚴重系統卡頓問題。
